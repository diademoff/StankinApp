# StankinAppDatabase

CLI-утилита для создания и просмотра базы данных расписания.

---

## Быстрый старт

### Требования

- .NET 8.0 SDK
- JSON-файлы расписания (по одному файлу на группу, имя файла = название группы)

### Сборка

```bash
dotnet build
```

---

## Создание базы данных

```bash
dotnet run -- create \
  --json-path /путь/к/папке/с/json \
  --year 2026 \
  --db-save-path schedule.db
```

| Параметр         | Обязателен | Описание                                              |
|------------------|:----------:|-------------------------------------------------------|
| `--json-path`    | ✅         | Папка с JSON-файлами расписания                       |
| `--year`         | ✅         | Учебный год (используется при разборе дат)            |
| `--db-save-path` | ❌         | Путь для сохранения БД (по умолчанию `schedule.db`)   |

Пример для Windows:

```bat
StankinAppDatabase.exe create --json-path C:\schedules\json --year 2026
```

### Что происходит при создании

1. Утилита последовательно обрабатывает все `.json`-файлы из указанной папки.
2. Если строка расписания успешно распарсена — она сразу записывается в БД.
3. Если строка вызывает ошибку парсинга — запускается интерактивный диалог (см. ниже).
4. После завершения создаётся файл `schedule.db`.

---

## Просмотр базы данных

```bash
dotnet run -- read --db-path schedule.db
```

Откроется TUI-интерфейс с навигацией по группам, кабинетам и преподавателям.

---

## Что делать при ошибке парсинга

Иногда парсер не может автоматически разобрать строку расписания. Это происходит по двум основным причинам:

### Причина 1 — Предмет содержит точку в названии

Парсер использует точку как разделитель между полями. Если название предмета само содержит точку (например, «Технологии индустрии 4.0»), он «теряется» и неправильно разбивает строку.

### Причина 2 — Нестандартное ФИО преподавателя

Парсер ожидает формат «Фамилия И.О.» (например, «Иванов И.И.»). Если у преподавателя другой формат имени (иностранные имена, двойные фамилии и т.д.), парсинг ломается.

---

### Как выглядит диалог исправления

Когда парсер встречает проблемную строку, он **останавливается** и показывает следующее:

```
┌────────────────────────────────────────────────────────────┐
│  ⚠  Не удалось распарсить строку расписания                │
│  Группа: ИДБ-22-01                                         │
└────────────────────────────────────────────────────────────┘

  Исходная строка:
  > Технологии индустрии 4.0. Юсеф Фарах. лекции. 305А. [10.09-15.10 ч.н.]

  Парсер попытался извлечь следующие занятия (с ошибками):
  [1]
      Предмет:      Технологии индустрии 4
      Преподаватель:
      Тип:          0

  Возможные причины ошибки:
  • Название предмета содержит точку (парсер принял её за разделитель).
  • ФИО преподавателя не в стандартном формате «Иванов И.И.».

  Выберите тип ошибки или пропустите строку:

  [1] Предмет с точкой в названии
  [2] Нестандартное ФИО преподавателя
  [s] Пропустить эту строку

  Ваш выбор:
```

### Что вводить

**Если проблема в предмете (`1`):**

```
  Название предмета: Технологии индустрии 4.0
```

Введите полное правильное название предмета **с точкой** — именно так, как оно должно выглядеть в итоге.

---

**Если проблема в преподавателе (`2`):**

```
  Неверное ФИО: Юсеф Фарах
  Правильное ФИО:             ← можно оставить пустым, тогда имя останется как есть
```

Введите имя **так, как оно стоит в исходной строке** (неверное), затем — как его нужно сохранить в базе.

---

**Пропустить строку (`s`):**

Строка не будет добавлена в базу данных. Это безопасно — вы сможете добавить её вручную позже или пересоздать БД с правильными исправлениями.

---

### Где хранятся исправления — файл `fallback_info.json`

Все исправления автоматически сохраняются в файл `fallback_info.json` рядом с утилитой:

```json
{
  "SubjectNamesWithDots": [
    "Технологии индустрии 4.0"
  ],
  "AnomalyTeachers": [
    {
      "IncorrectName": "Юсеф Фарах",
      "CorrectName": "Юсеф Фарах"
    }
  ]
}
```

При следующем запуске `create` утилита **автоматически** применит сохранённые исправления и не будет переспрашивать одно и то же.

> **Совет:** Если вы заранее знаете проблемные предметы или преподавателей — можно добавить их в `fallback_info.json` вручную до запуска, и диалогов не будет вообще.

---

### Если вы допустили ошибку при вводе

Откройте `fallback_info.json` в любом текстовом редакторе, исправьте нужную запись и пересоздайте базу:

```bash
dotnet run -- create --json-path /путь/к/json --year 2026 --db-save-path schedule.db
```

Старый файл `schedule.db` будет перезаписан.

## Формат JSON-файлов расписания

Файл должен называться по имени группы (например, `ИДБ-22-01.json`) и иметь структуру:

```json
{
  "2026-09-10": {
    "08:30-10:05": [
      "Математика. Иванов И.И. лекции. 305А. [10.09-15.10 ч.н.]"
    ]
  }
}
```

Формат строки занятия:

```
Предмет. Преподаватель И.О. тип_занятия. (подгруппа.) кабинет. [даты]
```

Допустимые типы занятий: `лекции`, `семинар`, `лабораторные занятия`, `Лекция`, `Семинар`, `Лабораторная`.

Формат дат: `дд.мм` или `дд.мм-дд.мм ч.н.` / `дд.мм-дд.мм к.н.` (через неделю / каждую неделю).

## Структура проекта

| Файл                  | Назначение                                              |
|-----------------------|---------------------------------------------------------|
| `Program.cs`          | Точка входа, CLI-команды `create` и `read`              |
| `DatabaseScraper.cs`  | Обход JSON-файлов и наполнение БД                       |
| `ScheduleJsonReader.cs` | Разбор JSON и парсинг строк расписания               |
| `ParserFallback.cs`   | Интерактивный диалог и хранилище исправлений            |
| `ErrorParsingInfo.cs` | Структура с данными о проблемной строке                 |
| `UI.cs`               | TUI-интерфейс просмотра расписания                      |
| `fallback_info.json`  | Сохранённые исправления (создаётся автоматически)       |
| `schedule.db`         | SQLite-база данных (создаётся командой `create`)        |
